# DRISHTI Technical Architecture and System Guide

This document describes the repository as implemented in code. It is intended for engineers who need to run, maintain, extend, review, or operate the system.

DRISHTI (Dynamic Resource Intelligence for Smart Highway and Traffic Intervention) is a Bengaluru traffic-operations platform. It combines public complaint intake, police access control, field-personnel tracking, dispatch management, event-impact prediction, resource recommendations, maps, and deployment chat.

## 1. System at a glance

The running application has four primary components:

1. A Next.js 15 and React 19 browser application in `frontend/`.
2. A FastAPI application (DRISHTI API version 2.0.0) in `backend/app/`.
3. A PostgreSQL database, commonly hosted by Supabase, created by `backend/migrations/`.
4. Redis for JWT revocation. Redis is optional at runtime but required for server-side logout invalidation.

The backend also performs local inference using scikit-learn artifacts under `backend/app/models/`. Mappls/MapMyIndia, SMTP, and a local Ollama server are optional integrations.

```text
Citizen browser                         Police browser
      |                                      |
      +------------ Next.js UI --------------+
                         |
                HTTPS/JSON + WebSocket
                         |
                     FastAPI
        +----------------+-------------------+
        |                |                   |
   PostgreSQL       Local ML models      External services
   operational      duration, impact,    Mappls, SMTP,
   state/history    resources, learning  optional Ollama
        |
      Redis (JWT blacklist; separate from PostgreSQL)
```

The browser never accesses PostgreSQL directly. All application data and authorization decisions pass through FastAPI. Model inference is local and synchronous; no external ML service is required.

## 3. Runtime startup

### Backend

Uvicorn imports `backend/app/main.py`. FastAPI's lifespan handler loads:

- `duration_model.pkl` and `impact_model.pkl` through `PredictionService`;
- `resource_deployment_model.pkl` and `learning_priority_model.pkl` through `ResourceRecommendationService`;
- the corresponding JSON feature and target manifests.

Failure to load the duration or impact model raises `PredictionError` during startup and prevents the backend from becoming ready. Failure to load the resource or learning models is intentionally absorbed; those recommendations fall back to deterministic policy rules.

The API exposes OpenAPI at `/docs`, `/redoc`, and `/openapi.json` through FastAPI defaults.

### Frontend

Next.js reads `NEXT_PUBLIC_API_URL` at build time. `frontend/src/lib/api.ts` creates one Axios client and attaches `Authorization: Bearer <token>` from browser local storage to each request.

The frontend is predominantly client-rendered because authentication and live operational data depend on browser APIs, local storage, geolocation, and map SDKs.

## 4. Main end-to-end workflows

### 4.1 Account registration and approval

1. An officer or reviewer submits `/register` or `/register/viewer`.
2. The frontend calls `POST /auth/register`.
3. The backend lowercases the email, bcrypt-hashes the password, and inserts an inactive `app_users` row with `approval_status = pending`.
4. An admin reviews the request at `/dashboard/access`.
5. Approval activates the account and, when a badge ID exists, upserts the user into `police_personnel` so the person can be dispatched.
6. Rejection keeps the account inactive and stores a rejection reason.
7. Approval/rejection email is attempted through SMTP. Email failure is logged and does not roll back the account decision.

All requested roles, including `admin`, enter the same approval workflow. There is no seed-user creation in the current backend code; an initial admin must therefore be inserted or activated operationally in the database.

### 4.2 Login, authorization, and logout

1. `POST /auth/login` verifies approval, active status, and the bcrypt password.
2. The backend issues a signed JWT containing `sub`, `email`, `role`, `jti`, `iat`, and `exp`.
3. The frontend stores the JWT as `drishti_access_token` and the user snapshot as `drishti_user` in local storage.
4. `ProtectedRoute` calls `GET /auth/me` on navigation. The backend validates the signature, expiration, Redis blacklist, and current database user state.
5. The frontend redirects unauthorized roles to their configured home dashboard.
6. Logout calls `POST /auth/logout`, blacklists the token's JTI in Redis until token expiry, and always clears local storage.

If Redis is missing or unavailable, login and authenticated requests still work. Logout then only clears the browser session; a copied JWT remains valid until expiry or the user is deactivated.

### 4.3 Citizen complaint intake and tracking

1. A citizen opens `/citizen/grievance`, selects a complaint type, enters a location and description, and may share browser coordinates.
2. `POST /citizen/grievances` is public.
3. If coordinates are absent and MapMyIndia is configured, the backend geocodes a query assembled from the location, corridor, zone, and Bengaluru.
4. `triage_grievance` derives severity, a 0-100 priority score, and a rule-based dispatch recommendation. It invokes the event prediction model where possible and may ask Ollama to rewrite the recommendation into two operational sentences.
5. The complaint is stored with a tracking token in the form `DRS-BTP-XXXXXXXXXX`.
6. An `agent_actions` audit row records the triage recommendation.
7. The citizen can later call the public `GET /citizen/grievances/{tracking_id}` through `/citizen/track`.

Complaint submission requires PostgreSQL. Geocoding and Ollama are optional; both fail back without preventing submission.

### 4.4 Complaint review and event forecasting

Police roles can read the latest 50 grievances. The admin complaint page refreshes every 30 seconds and can move complaints through the operational statuses.

The admin can expand a selected complaint into `PredictionForm`. Location, zone, corridor, and complaint details seed the event request. `POST /predict-impact` then:

1. validates the request with Pydantic;
2. creates duration and impact feature frames in the exact persisted feature order;
3. predicts log-duration and converts it back with `expm1`;
4. classifies impact as Low, Medium, High, or Critical;
5. extracts keywords, risks, and urgency with the local NLP rules;
6. predicts a resource/rank/equipment plan and learning priority, or uses policy fallback;
7. returns the combined response;
8. best-effort stores the request, response, operator override, and model signals in `prediction_events`.

Prediction persistence is deliberately non-blocking. If PostgreSQL logging fails, the model result still reaches the operator.

### 4.5 Deployment creation and nearest-officer assignment

Admins and operators use `DeploymentAssignmentPanel` to create an order for a grievance.

1. The API loads the grievance and creates an order number such as `DRS-DEP-XXXXXXXXXX`.
2. The caller may provide personnel IDs manually.
3. If auto-assignment is enabled and no IDs were supplied, the service filters available active personnel and ranks them using Haversine distance from the complaint coordinates. Personnel without usable GPS are ranked after located personnel.
4. It inserts `deployment_orders` and junction rows in `deployment_order_personnel`.
5. Assigned personnel become unavailable.
6. A WhatsApp-shaped notification payload is stored with the order, but this repository does not actually send a WhatsApp message.

The created order is visible to the assigned officer because field assignment lookup joins the logged-in user's badge ID to `police_personnel` and then to deployment assignments.

### 4.6 Field-officer operation

The `operator` role maps to `/dashboard/field` and represents a field officer in the current UI.

- Assignments are refreshed every 30 seconds.
- When duty location is enabled, `navigator.geolocation.watchPosition` sends the officer's current position to `POST /field/me/location`.
- Position writes update latitude, longitude, accuracy, and `last_location_at` by badge ID.
- The field page can progress an order through `enroute`, `onscene`, `resolved`, or `escalated`.
- Officers can submit a field report through `POST /field/me/report`; reporter identity is filled from the JWT user.
- The en-route panel fetches a server-proxied Mappls route and shows nearby personnel.

When a deployment becomes `resolved`, assigned personnel are marked available and the linked grievance is moved to `pending_verification`. An admin must separately mark the grievance `resolved`; this is the system's two-step resolution control.

### 4.7 Maps and routing

The UI supports two mapping paths:

- Mappls browser SDK components (`PersonnelMap`, `MapplsRouteMap`) for operational maps.
- React Leaflet/OpenStreetMap (`RouteMapLeaflet`) as a UI fallback.

`GET /field/route` keeps the Mappls API key on the server and requests GeoJSON route geometry. If the provider request fails after an API key was found, it returns a straight line with `fallback: true`. If no key exists, it returns HTTP 503, and the frontend can use its map fallback.

Geocoding uses the separately configurable `MAPMYINDIA_GEOCODE_URL` and the same API key.

### 4.8 Deployment chat

Each deployment ID is a chat room.

1. The browser opens `/ws/chat/{deployment_id}?token=<JWT>`.
2. The backend authenticates the query-string JWT and accepts the socket.
3. It sends up to 100 persisted messages as a `history` frame.
4. New non-empty messages are inserted into `deployment_chat_messages` and broadcast to sockets in the room.
5. The frontend hook reconnects after a dropped connection.

History is durable in PostgreSQL, but live fan-out is an in-process dictionary. With the Docker backend configured for two Uvicorn workers, clients connected to different workers do not receive each other's live broadcast until history is reloaded. Production multi-worker or multi-instance chat needs a Redis pub/sub or comparable shared backplane.

## 8. Machine-learning architecture

### 8.1 Active inference models

The prediction request's core model features are:

- event cause and event type;
- priority and road-closure flag;
- corridor and zone;
- latitude and longitude;
- hour, day of week, and month.

The duration model is a scikit-learn pipeline trained on `log1p(duration_minutes)`. At inference the service applies `expm1` and rounds to two decimals. The impact classifier directly predicts Low, Medium, High, or Critical.

The resource model consumes the base event features plus predicted duration, impact level, NLP urgency, risk count, and estimated crowd size. It predicts personnel totals and rank composition, barricades, tow units, medical units, and diversion confidence. Values are clamped to operational bounds.

The learning model predicts retraining priority. Every response asks for post-event ground truth such as actual duration, actual impact, personnel adequacy, and diversion performance.

### 8.2 Fallback behavior

If the resource or learning artifact cannot load or inference throws, policy logic computes a recommendation from duration, impact multiplier, crowd size, closure status, and urgency. Duration and impact inference have no equivalent fallback: if either active model is unavailable, `/predict-impact` fails.

### 8.3 Weekly retraining

`backend/ml/weekly_retraining_pipeline.py`:

1. reads the anonymized Astram CSV;
2. derives completion time and duration;
3. removes unusable rows and creates log-duration;
4. appends eligible rows from the `retraining_prediction_dataset` database view;
5. performs an 80/20 split;
6. trains 300-tree random-forest regression and classification pipelines;
7. records MAE/R2 on log-duration and classifier accuracy;
8. writes timestamped candidate artifacts to `backend/app/models/candidates/`;
9. also trains candidate resource-deployment and learning-priority models.

Impact labels in this pipeline are derived from duration thresholds: Low <= 72 minutes, Medium <= 793, High <= 17,146, and Critical above that.

Candidate generation does not activate a model. No code copies a candidate into the active filenames, updates `model_registry`, marks retraining rows used, or hot-reloads the API. Promotion is a manual operational step and requires a backend restart.

## 9. Database design

Migrations must be applied in numeric order from `001` through `010`.

### Core tables

| Table | Role |
|---|---|
| `app_users` | Accounts, role, password hash, approval state, badge/rank/unit metadata |
| `police_personnel` | Dispatchable personnel, availability, contact, GPS, active flag |
| `citizen_grievances` | Reporter input, triage, coordinates, tracking and operational status |
| `prediction_events` | Prediction inputs/outputs, feedback, model version and retraining eligibility |
| `deployment_orders` | Complaint-linked field orders, location, recommendation, notification payload and status |
| `deployment_order_personnel` | Many-to-many assignment junction |
| `deployment_chat_messages` | Durable deployment-room chat; lazily created by the service rather than a numbered migration |

### Audit, integration, and ML tables

| Table/view | Role |
|---|---|
| `drishti_event_log` | Append-oriented aggregate event history populated by triggers and repository writes |
| `agent_actions` | Agent recommendation audit records |
| `event_stream_outbox` | Transactional-outbox-shaped queue for a future publisher |
| `event_conflict_escalations` | Schema for overlapping corridor/junction review |
| `event_pii_vault` | Separation point for sensitive model-excluded data |
| `retraining_runs` | Retraining run metadata |
| `model_registry` | Model artifact metadata and active flag |
| `retraining_prediction_dataset` | Ground-truth-ready prediction view for training |

Some schema elements represent planned architecture rather than wired runtime behavior. The application does not currently publish the outbox to Kafka, use `event_conflict_escalations`, write PII into `event_pii_vault`, or use `model_registry` to select active artifacts.

### Principal relationships

```text
app_users 1 ---- 0..1 police_personnel (linked operationally by badge_id)
app_users 1 ---- * deployment_orders (commander_user_id)
citizen_grievances 1 ---- * deployment_orders
prediction_events 1 ---- * deployment_orders (schema supports it; create-order does not set it)
deployment_orders * ---- * police_personnel (deployment_order_personnel)
deployment_orders 1 ---- * deployment_chat_messages (text deployment_id, no FK)
```

### Status lifecycles

Complaint contracts use:

```text
pending -> in_progress -> assigned -> pending_verification -> resolved -> closed
```

Deployment contracts use:

```text
draft -> issued -> enroute -> onscene -> resolved
                              \-> escalated
any operational point may be cancelled by an authorized caller
```

The API validates allowed status values but does not enforce a strict transition graph. An authorized caller can generally jump between valid values. `notes` fields in both status-update requests are accepted but are not persisted by the current services.

## 10. Middleware, resilience, and observability

The backend installs:

- CORS for `ALLOWED_ORIGINS`, defaulting to localhost ports 3000 and 3001;
- request IDs from `X-Request-ID` or a generated UUID;
- `X-Response-Time` timing;
- basic security headers;
- an in-memory per-IP rate limit of 120 HTTP requests per 60 seconds;
- generic exception handling with a special PostgreSQL-unavailable response.

The rate limiter is per process, resets on restart, does not coordinate between workers, trusts the direct socket IP, and does not cover WebSocket messages. It is a basic safeguard, not a distributed production rate limiter.

Resilience is intentionally asymmetric:

- prediction logging, Redis revocation, SMTP, geocoding, Ollama, and resource-model inference degrade gracefully;
- primary PostgreSQL workflows (auth, complaints, dispatch, chat) require the database;
- duration/impact models are startup-critical;
- operations summary returns zeroed data on database failure rather than an error.

Logs go to standard output through Python logging. There is no metrics exporter, tracing backend, error-monitoring SDK, structured JSON logging, or health check for PostgreSQL/Redis/Mappls.

## 12. Local development and deployment

### Native development

Apply the SQL migrations to PostgreSQL, then create `backend/.env`.

```powershell
cd backend
python -m pip install -r requirements.txt
python -m uvicorn app.main:app --reload --port 8000
```

In another terminal:

```powershell
cd frontend
npm install
$env:NEXT_PUBLIC_API_URL = "http://localhost:8000"
npm run dev
```

Useful checks:

```text
Frontend:       http://localhost:3000
Backend health: http://localhost:8000/health
OpenAPI:        http://localhost:8000/docs
Redis:          localhost:6379
```

### Docker Compose

```powershell
docker compose up --build
```

Compose starts Redis, a two-worker Uvicorn backend on port 8000, and the standalone Next.js server on port 3000. It expects `backend/.env` to exist. It does not start PostgreSQL, Ollama, Kafka, or an SMTP server.

The frontend image receives `NEXT_PUBLIC_API_URL=http://localhost:8000` at build time. For a non-local deployment, rebuild with the externally reachable API URL; the browser cannot use a Docker-only service hostname.

## 13. Current implementation risks and extension points

The following are the most important findings for production hardening:

1. Protect or remove the public badge-location endpoint.
2. Authorize chat and chat history at deployment-membership level, not only JWT/role level.
3. Replace in-process WebSocket fan-out with Redis pub/sub for multi-worker consistency.
4. Add a numbered migration for `deployment_chat_messages` and a foreign key where practical.
5. Establish a bootstrap-admin process; current registration makes every new account pending.
6. Use a PostgreSQL connection pool and explicit transaction/error policies for core workflows.
7. Enforce state-transition rules and persist status-update notes.
8. Implement an actual notification adapter if WhatsApp delivery is expected.
9. Wire outbox publishing, conflict detection, and PII-vault usage or clearly remove them from the active architecture claim.
10. Add automated tests. No backend or frontend test suite is present in this repository.
11. Add a candidate evaluation/promotion process and model rollback procedure.
12. Align stale documentation and root artifacts with the active `backend/app/models` runtime layout.

Other notable behavior:

- JWTs are stored in local storage, so browser XSS would expose them. A hardened deployment should evaluate secure, HttpOnly cookie sessions and a Content Security Policy.
- WebSocket JWTs appear in the URL query string and may be captured by infrastructure logs.
- The field-location utility route is not wrapped in frontend auth and the corresponding badge endpoint is public.
- `event_conflict_escalations` and the transactional outbox are schemas only; they do not yet enforce the stated operational policies.
- Chat schema initialization assumes a usable database URL and `gen_random_uuid()` support.
- Root-level model files are not loaded by the API, which can confuse manual model promotion.

## 15. Verification checklist

After a change, verify at minimum:

1. `GET /health` reports all expected models loaded.
2. Registration remains pending and admin approval enables login.
3. Role redirects match backend authorization.
4. A citizen complaint receives a tracking ID and can be looked up.
5. Prediction returns duration, impact, NLP, resources, and learning signals.
6. A deployment assigns personnel and changes their availability.
7. The assigned badge sees the order and can publish GPS.
8. Resolving an order moves its grievance to pending verification.
9. Admin resolution completes the two-step flow.
10. Route fallback behaves acceptably when Mappls is unavailable.
11. Chat history survives reconnect and live chat works across the intended deployment topology.
12. Database-unavailable behavior matches whether the workflow is core or best-effort.

This architecture is a functioning operational prototype with thoughtful fallback behavior around optional intelligence services. Its central path is PostgreSQL-backed and synchronous, while ML inference remains local. The clearest next engineering boundary is turning the current single-service prototype assumptions—especially chat fan-out, connection management, device location trust, and manual model promotion—into explicit production infrastructure.
